# レジの助 (reji-no-suke)

GDGoC Osaka のハッキングハンズオンイベント用に作成された「居酒屋ぼったくり調査」Webアプリケーションです。

## 概要
参加者がブラウザのDevTools（NetworkタブやConsole）を駆使して、隠された真実（正しい注文金額）を暴き出すことを目的とした学習用アプリです。
意図的に**JWTトークン漏洩**や**IDOR（安全でない直接オブジェクト参照）**の脆弱性を仕込んでおり、実際の商用アプリでも起こりうるセキュリティミスを体験できます。

今回は**「モバイルオーダーアプリ（客が自身のスマホで注文するシステム）」**の形をとり、客として自らハッキングを行うシナリオとなっています。

## 環境構築と起動手順

本アプリは Node.js と SQLite（ファイルベース）で動作します。

### 1. 依存関係のインストール
```bash
npm install
```

### 2. データベースの初期化とシードデータ投入
```bash
npm run seed
```
※ 初期データ（テーブル、メニュー一覧など）が `database.sqlite` に作成されます。データをリセットしたい場合はいつでもこのコマンドを実行してください。

### 3. サーバー起動
```bash
npm start
```
サーバーが `http://localhost:3000` で起動します。

## ハッキングの想定シナリオと攻略手順

**シナリオ**：「居酒屋の席にあるQRコードからオンライン注文した。しかし、お会計画面を見ると頼んだ覚えのない金額（＋2500円）が加算されている！このシステムの裏側をハッキングして、本当のお会計データを暴き出せ。」

### 参加者の行動フロー（攻略の答え）

1. **システムの利用開始**
   - `http://localhost:3000` にアクセスし、卓番号（例: `table_3`）を入力して注文画面に進みます（QRコード読み取りの代わり）。
   
2. **注文とボッタクリの確認**
   - いくつか商品を注文し、「お会計確認」タブを開きます。
   - 頼んだ商品の小計より、合計金額が明らかに高いことに気づきます。

3. **エンドポイントと認証トークンの特定**
   - DevTools (F12) の Network タブを開きながら、ページをリロードするか再度ログインします。
   - `POST /api/v2/auth/signin` のレスポンスボディの中に、意図せず平文で含まれている `token`（JWT）を発見します。

4. **データの傍受（IDOR脆弱性）**
   - 「お会計確認」タブを開いた際に叩かれている `GET /api/v2/r/session-data` のレスポンスを確認します。
   - 画面上は自分の卓の履歴しか見えませんが、サーバー側の認可不備（IDOR）によりレスポンスデータには**全卓の注文データが丸見え**になっていることに気づきます。

5. **Base64のデコード**
   - 漏洩したデータ内の `table_id` が `dGFibGVfMw==` のように不自然な文字列であることに気づきます。
   - ブラウザのConsoleで `atob('dGFibGVfMw==')` を実行し、自分（またはターゲット）の卓番号（例: `table_3`）を特定します。

6. **ゴール：正しいお会計の抽出**
   - 「特定の卓番号を指定すると、正しい合計金額を綺麗に返してくれる隠しAPI（本来はレジ連携等で内部利用するもの）」が存在します。
   - 参加者はConsoleタブで以下のコマンドを直接実行（fetch）し、データベースから直接情報を引っこ抜きます。
   
   ```javascript
   fetch('/api/v2/orders/table_3').then(r => r.json()).then(console.log)
   ```
   
   - レスポンスとして「ハッキング成功！」のメッセージと共に、**ボッタクリ加算されていない正しい合計金額**と明細が返ってくればクリアです。
